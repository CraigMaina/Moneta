## 2026-07-14 â€” Phase 3: `parse-sms` Edge Function + `parse_misses` miss-log (backend-engineer)

**Scope:** `supabase/migrations/20260714100000_create_parse_misses.sql` (new); `supabase/tests/database/parse_misses.test.sql` (new); `supabase/tests/database/rls.test.sql` (extended â€” `plan(25)` â†’ `plan(26)`, added a `has_table` assertion for `parse_misses`); `supabase/functions/parse-sms/{deno.json,schema.ts,prompt.ts,index.ts,schema.test.ts}` (new); `eslint.config.js` (one new override block for `supabase/functions/**`). No files under `src/parser/` or `src/features/` touched, per brief.

- **`parse_misses.raw_sms_hash` gets a *partial* unique index (`where raw_sms_hash is not null`) even though the column is `not null`.** The brief explicitly asked for a partial unique index "mirroring `transactions.mpesa_ref`'s convention" â€” but `mpesa_ref` is partial *because* it's nullable (manual entries have no SMS), while `raw_sms_hash` here is always populated by the Edge Function. Kept the `where` clause anyway purely so the index reads the same way as the established convention at a glance and stays correct for free if `raw_sms_hash` is ever loosened to nullable later â€” a no-op today, not a functional deviation.
- **`resolved` has no automatic flip mechanism in this phase.** The column exists per the brief (a bookkeeping flag for "a deterministic pattern now covers this shape") but nothing in this slice sets it to `true` â€” that's manual/offline curation by whoever authors new `patterns.json` entries from this log, not app logic. Recorded so a future phase doesn't assume it's already wired.
- **The Edge Function client never uses the service-role key.** `index.ts` verifies the caller's JWT by creating a `supabase-js` client with the **anon key** plus the caller's own `Authorization` header forwarded (`global.headers.Authorization`), then calls `supabase.auth.getUser()`. Every subsequent DB write (the `parse_misses` upsert) runs under that user's own session, so ordinary `user_id = auth.uid()` RLS policies apply exactly as they do to any client request â€” no bespoke bypass, no need to ever hold or reach for the service-role key in this function. `GEMINI_API_KEY` is the only genuinely function-private secret, read only via `Deno.env.get` and never echoed to the client.
- **LLM provider: Google Gemini (`gemini-2.0-flash`), free tier â€” replaced the initial Anthropic implementation (user directive: "we are not using anthropic â€¦ get something free").** The deterministic on-device parser handles the bulk of messages entirely offline; only genuine misses reach this function, so a free, fast model is the right fit, and a strict downstream validator never trusts the output blindly anyway. Key is free from https://aistudio.google.com/apikey; secret name `GEMINI_API_KEY`.
- **Gemini uses structured output (`generationConfig.responseSchema` + `responseMimeType: 'application/json'`), not free-text JSON in a prompt.** The model returns an envelope `{ parseable: boolean, transaction: <fields|null> }` â€” `parseable: false` (or a null transaction) is the "this isn't an M-PESA message" sentinel. `responseSchema` (an OpenAPI subset in `prompt.ts`) only *guides* the shape; it can't express the contract's cross-field invariants (transfer â‡’ hint+direction, category-matches-kind, reversal â‡’ reversalOfRef), so the **real, only-trusted gate** is `schema.ts`'s hand-mirrored `parsedMpesaMessageSchema.superRefine`, run on `interpretModelJson(jsonText, raw)` after every call. ## 2026-07-14 â€” `handle_new_user` signup trigger (backend-engineer)

**Scope:** `supabase/migrations/20260714090000_create_handle_new_user_trigger.sql`, `supabase/tests/database/new_user.test.sql`. No app code touched; `supabase/seed.sql` left as-is per the task brief.

- **SECURITY DEFINER + `set search_path = ''`, every reference fully schema-qualified.** The trigger fires on `auth.users` insert, before the new user has a JWT/session, so `auth.uid()` is null and the ordinary `user_id = auth.uid()` RLS policies on `profiles`/`accounts`/`categories` would reject every insert. Running as the function owner (the migration role, which owns these tables and â€” per Postgres's default RLS semantics â€” is not subject to their own RLS policies unless `FORCE ROW LEVEL SECURITY` is set, which it isn't here) is what lets the seed succeed at all; this is the standard Supabase idiom for a signup trigger, not a bespoke bypass. Locking `search_path` to empty (rather than leaving it at `public`) closes the classic SECURITY DEFINER hijack vector â€” an attacker who could get a same-named object earlier in a non-empty search_path resolved instead of the intended `public.*` table/type. Every insert hardcodes `user_id = new.id` (the row literally being created in `auth.users`), so this trigger structurally cannot seed data under any user but the one just created.
- **Seed data is a byte-for-byte mirror of `supabase/seed.sql`'s dev-user seed** (3 accounts: M-PESA/mpesa/smartphone, Cash/cash/banknote, Bank/bank/landmark; 20 categories: 5 income + 15 expense, exact name/kind/icon/color/sort_order) â€” verified programmatically (not just by eye) that both files produce identical tuples. `seed.sql` stays local-dev-only (fixed synthetic user, runs only on `db reset`); this trigger is what actually seeds every *real* signup going forward. The two are deliberately duplicated rather than one calling the other, since `seed.sql` seeds a specific fixed user id via a `do $$ ... $$` block and isn't structured as a reusable function â€” a future refactor could extract a shared `seed_default_data(user_id)` helper if this duplication becomes a maintenance burden, but for now the mirror is small and explicit.
- **`on conflict (user_id, name) do nothing` (accounts/categories) / `on conflict (user_id) do nothing` (profiles)**, matching the exact unique constraints already in the accounts/categories/profiles migrations. This makes the trigger resilient to a re-fire (e.g. Supabase Auth retrying an insert) or a profile/accounts row that already exists for some other reason â€” it never errors, never duplicates.
- **`display_name` reads `new.raw_user_meta_data ->> 'name'`, falling through to `null` when absent** â€” per the task's explicit instruction. Email/password signups typically won't populate this; a future Google OAuth signup may populate `name` (or `full_name`, not read here) in `raw_user_meta_data` â€” revisit the exact key once Google sign-in is wired if it turns out to use a different metadata key.
- **pgTAP test (`new_user.test.sql`, plan 5)** inserts a synthetic `auth.users` row inside `begin/rollback` (same proven pattern as `rls.test.sql`'s synthetic users) and asserts: exactly 1 `profiles` row, exactly 3 `accounts` rows, exactly 20 `categories` rows, plus a spot-check each for `M-PESA` and `Food & Groceries`. Self-rolling-back â€” leaves no residue on whatever database (including cloud) it runs against.
- **Not run against cloud from this environment** (Docker-free constraint per prior entries) â€” migration was cross-checked by eyeball plus the programmatic seed-data-equality check above; `npm run check` (179 tests, unaffected â€” no app code touched) confirmed green. The pgTAP execution and `supabase db push` are for the lead to run via `SUPABASE_DB_URL=â€¦ node supabase/tests/run-pgtap.mjs supabase/tests/database/new_user.test.sql`.

## 2026-07-13 â€” Phase 2: typed data layer, transaction feature folder (feature-engineer)

**Scope:** `src/features/transactions/{types,queryKeys,nairobiDate,queries,mutations,schemas,balanceDelta,useSafeToSpend}.ts` + colocated tests, `src/features/transactions/hooks/useAuthUserId.ts` (+ test), `src/test/supabaseTestHelpers.ts`. This is deliberately visible-and-wrong-until-onboarded rather than invisible-and-stuck â€” matches "never ship a screen without loading/empty/error states" in spirit, since a perpetual loading spinner isn't a real state either.

- **`updateTransactionSchema`'s cross-field invariants (transfer â‡” `counter_account_id`, transfer â‡’ no `category_id`) only fire when the *same patch* carries the relevant fields together.** A patch is a partial object; you can't validate "transfer requires a counter_account_id" against a patch that only touches `note` without first fetching the row being patched (which the schema layer deliberately doesn't do â€” zod validates shape, not database state). Concretely: changing `kind` to `'transfer'` without also patching `counter_account_id` in the same call is rejected; patching only `counter_account_id` on an existing (already-valid) transfer is allowed through since the schema can't see the untouched `kind`. - **A `crypto.randomUUID()`-based `optimistic-<uuid>` id, not a monotonic counter or `nanoid`,** for the placeholder row an add-mutation inserts into the cache before the server assigns a real id. `crypto.randomUUID` is a Node â‰¥19/all evergreen-browser global â€” no new dependency, no collision risk with real (Postgres `gen_random_uuid()`-style) ids. `isOptimisticId(id)` is exported so a future component can show a pending affordance (e.g. a subtle opacity) on rows not yet confirmed by the server.

- **Balance optimism is delta-based (`balanceDelta.ts`), computed identically for add/update/delete, not "refetch `account_balances` synchronously."** `useUpdateTransaction`/`useDeleteTransaction` reverse the *old* row's delta (found in the cached transaction list, if present) before applying the *new* one â€” this exactly mirrors what the `account_balances` SQL view would compute from the full transaction set, so the optimistic number a user sees mid-flight matches what the server will confirm on `onSettled` invalidation, rather than drifting. `fee_cents` is never touched by this math, matching `backend-engineer`'s existing "fee-modeling decision" entry (a fee is its own separate `expense` row). If the row being updated/deleted isn't in any cached list yet (e.g. a cold cache), the balance patch is skipped rather than guessed â€” the `onSettled` invalidation still corrects it from the server.

- **Test fake for supabase-js's query builder (`src/test/supabaseTestHelpers.ts`) is a `Proxy`-based `chainable()` with zero dependency on `vi`/mocking libraries**, so it's reusable as a plain value even outside `vi.mock` factories. - **`vi.hoisted(() => ({ auth: {...}, from: vi.fn() }))` is duplicated (not shared) across `queries.test.tsx`/`mutations.test.tsx`/`useSafeToSpend.test.tsx`/`useAuthUserId.test.tsx`**, rather than importing one shared hoisted mock object. `vi.hoisted`'s callback runs before other imports are evaluated within that file, so an imported factory function can't reliably be called from inside it without risking a "used before initialization" hazard across files/bundlers â€” the few-line duplication is the safer, standard-idiom trade-off here (Vitest's own docs recommend inlining the hoisted factory).

## 2026-07-13 â€” Phase 2 start: safe-to-spend calculator semantics (lead)

`src/lib/safeToSpend.ts` â€” the pure, exhaustively-tested hero calculator (19 tests; the master prompt reserves the money path for the lead). Two semantic decisions the PRD formula left implicit:

## 2026-07-14 â€” Phase 3 M-PESA parser core (parser-engineer)

**Scope:** `src/parser/` only (`types.ts`, `money.ts`, `timestamp.ts`, `merchant.ts`, `patterns.json`, `patternTable.ts`, `index.ts`, `__fixtures__/`, and their colocated `*.test.ts`), plus a one-line `tsconfig.app.json` compiler-flag addition (see below). - **Pochi la Biashara vs. plain "sent to person" is a documented, real-world ambiguity â€” not fully solved.** On the PAYING customer's own confirmation SMS, a Pochi la Biashara payment and a normal person-to-person send are usually textually identical (Safaricom doesn't consistently brand Pochi payments differently on the sender side). ## Habit engine â€” logging streak (F8)

## Rule-based nudges (F9)

- **No budgets table exists, so "80%-of-category-pace" baselines against the category's own trailing-3-month average**, not a set budget. ## App lock + delete-all-data (F11)

- **The PIN is a *local convenience* gate on top of the real boundary (Supabase auth + RLS), never a replacement.** It's device-local: we store only a PBKDF2-SHA-256 hash (210k iterations, per-device random salt) in IndexedDB and verify with a constant-time compare. The raw PIN is never stored or transmitted. We deliberately do NOT use the `profiles.pin_hash` server column â€” a synced PIN would leave the device and add no real security over auth. `profiles.pin_hash` stays null (and is nulled by delete-all).
- **PINs are fixed at 4 digits.** Universal, familiar, and it makes both setup (choose â†’ confirm) and unlock (auto-submit on the 4th digit) trivial â€” no length ambiguity, no submit button. `PinPad` fires `onComplete` from the keypress handler so verification runs in an event handler, not a render effect.
- **Auto-lock on background:** `LockGate` locks whenever `visibilitychange` goes hidden, so returning to a foregrounded PWA always requires unlock. Simple and safe over a timeout for v1. The gate reads config once on mount (splash until known, like the other gates) and reacts to Settings changes via `lockStore.refresh()`.
- **Biometric unlock is optional WebAuthn platform-authenticator, capability-detected and fail-safe.** `isUserVerifyingPlatformAuthenticatorAvailable()` gates the toggle; a successful user-verifying assertion unlocks (local gate, so a random challenge with no server verification is acceptable). Every ceremony is wrapped so an unsupported/cancelled attempt is a clean `false` and the PIN path always remains. Not unit-tested (device-dependent) â€” kept minimal and guarded.
- **Forgotten-PIN escape = sign out + clear local lock.** Re-authenticating with the account is the genuine access control, so a forgotten PIN clears the device lock and signs out rather than stranding the user; the next sign-in starts lock-free.
- **"Delete all data" wipes financial *records* and resets the profile, but keeps the account/category scaffolding** (seeded at signup) so the app stays usable to start over â€” the type-to-confirm copy says exactly this, so it isn't misleading. Deletes are FK-ordered (goal_contributions before goals/transactions), RLS- and `user_id`-scoped; then it clears the query cache, the IndexedDB persister, and all lock data. `parse_misses` is excluded (parser telemetry, and not in the generated `Database` type). True auth-user deletion is a server/admin op, out of scope â€” noted for a future Edge Function.
- **Delete-all is guarded by type-to-confirm ("DELETE")** in a sheet that spells out what's removed and kept â€” an irreversible action never hides behind one tap (CLAUDE.md). The user performs it; on success the app resets to onboarding (profile consent cleared).

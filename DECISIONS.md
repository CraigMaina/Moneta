## 2026-07-14 ‚Äî Phase 3: `parse-sms` Edge Function + `parse_misses` miss-log (backend-engineer)

**Scope:** `supabase/migrations/20260714100000_create_parse_misses.sql` (new); `supabase/tests/database/parse_misses.test.sql` (new); `supabase/tests/database/rls.test.sql` (extended ‚Äî `plan(25)` ‚Üí `plan(26)`, added a `has_table` assertion for `parse_misses`); `supabase/functions/parse-sms/{deno.json,schema.ts,prompt.ts,index.ts,schema.test.ts}` (new); `eslint.config.js` (one new override block for `supabase/functions/**`). No files under `src/parser/` or `src/features/` touched, per brief.

- **`parse_misses.raw_sms_hash` gets a *partial* unique index (`where raw_sms_hash is not null`) even though the column is `not null`.** The brief explicitly asked for a partial unique index "mirroring `transactions.mpesa_ref`'s convention" ‚Äî but `mpesa_ref` is partial *because* it's nullable (manual entries have no SMS), while `raw_sms_hash` here is always populated by the Edge Function. A plain unique index would behave identically today. Kept the `where` clause anyway purely so the index reads the same way as the established convention at a glance and stays correct for free if `raw_sms_hash` is ever loosened to nullable later ‚Äî a no-op today, not a functional deviation.
- **`resolved` has no automatic flip mechanism in this phase.** The column exists per the brief (a bookkeeping flag for "a deterministic pattern now covers this shape") but nothing in this slice sets it to `true` ‚Äî that's manual/offline curation by whoever authors new `patterns.json` entries from this log, not app logic. Recorded so a future phase doesn't assume it's already wired.
- **The Edge Function client never uses the service-role key.** `index.ts` verifies the caller's JWT by creating a `supabase-js` client with the **anon key** plus the caller's own `Authorization` header forwarded (`global.headers.Authorization`), then calls `supabase.auth.getUser()`. Every subsequent DB write (the `parse_misses` upsert) runs under that user's own session, so ordinary `user_id = auth.uid()` RLS policies apply exactly as they do to any client request ‚Äî no bespoke bypass, no need to ever hold or reach for the service-role key in this function. `GEMINI_API_KEY` is the only genuinely function-private secret, read only via `Deno.env.get` and never echoed to the client.
- **LLM provider: Google Gemini (`gemini-2.0-flash`), free tier ‚Äî replaced the initial Anthropic implementation (user directive: "we are not using anthropic ‚Ä¶ get something free").** The deterministic on-device parser handles the bulk of messages entirely offline; only genuine misses reach this function, so a free, fast model is the right fit, and a strict downstream validator never trusts the output blindly anyway. Key is free from https://aistudio.google.com/apikey; secret name `GEMINI_API_KEY`.
- **Gemini uses structured output (`generationConfig.responseSchema` + `responseMimeType: 'application/json'`), not free-text JSON in a prompt.** The model returns an envelope `{ parseable: boolean, transaction: <fields|null> }` ‚Äî `parseable: false` (or a null transaction) is the "this isn't an M-PESA message" sentinel. `responseSchema` (an OpenAPI subset in `prompt.ts`) only *guides* the shape; it can't express the contract's cross-field invariants (transfer ‚áí hint+direction, category-matches-kind, reversal ‚áí reversalOfRef), so the **real, only-trusted gate** is `schema.ts`'s hand-mirrored `parsedMpesaMessageSchema.superRefine`, run on `interpretModelJson(jsonText, raw)` after every call. The API key travels in the `x-goog-api-key` header, never the URL query string (no secret in logs/URLs).
- **A Gemini API failure (network error, non-2xx, malformed/empty response, unparseable JSON) is treated identically to a validation failure**: caught, logged to `parse_misses` with `llm_succeeded: false`, and the function still returns `200 { status: 'manual', raw }` rather than a 5xx. Rationale: the client's only actionable response to *any* "couldn't get a confident parse" outcome is the same manual-entry prefill (CLAUDE.md: "on validation failure, surface manual-entry prefill, never a guess") ‚Äî a distinct 5xx would just make the client build a second error branch for no behavioral difference. The failure stays visible server-side via `console.error` and the `parse_misses` row. Genuine request-shape problems (missing/invalid `Authorization`, a malformed body, a missing `GEMINI_API_KEY` secret) DO return real 401/400/500s ‚Äî only the "we tried to parse and couldn't" case folds into `200 manual`.
- **`schema.ts`/`prompt.ts` are written as plain, runtime-agnostic TypeScript modules (bare `zod` import, no `Deno.*` calls) importable unmodified by both the Deno entrypoint and Vitest.** This is enabled by a per-function `supabase/functions/parse-sms/deno.json` import map (`{"imports": {"zod": "npm:zod@^4.4.3", "@supabase/supabase-js": "npm:@supabase/supabase-js@^2.110.2"}}`, the modern Supabase-documented convention) ‚Äî under Deno, the bare `zod` specifier resolves via that map; under Vitest/Node it resolves via the project's existing `node_modules/zod` (already a dependency). This is what let the brief's "lightweight test of the function's pure validation/mapping logic... runnable without network/secrets" land as an ordinary Vitest file (`schema.test.ts`, 8 tests) picked up automatically by `npm run test`/`npm run check`, rather than needing a separate Deno test runner or mocked Deno globals. `index.ts` (the only file that touches `Deno.serve`/`Deno.env`/network/DB) is intentionally NOT imported by any test ‚Äî the brief scoped testing to the pure core, and importing `index.ts` under Vitest would execute `Deno.serve(...)` as a side effect at module-load time, which is neither meaningful nor safe in that environment.
- **`eslint.config.js` gained one new override block** (`files: ['supabase/functions/**/*.ts']`, adding `globals.denoBuiltin` on top of the existing browser/node sets) ‚Äî needed because the shared `**/*.{ts,tsx}` block's `no-undef`-adjacent checks would otherwise flag `Deno`/`Deno.serve`/`Deno.env` as undefined globals (this repo's `globals` package ships a `denoBuiltin` set precisely for this). `tsconfig.app.json`'s `include: ["src"]` already excludes `supabase/` from `tsc -b`'s type-checked project, so this is purely a lint-scope fix, not a new type-checking surface; verified `npm run lint` is clean project-wide after the change.
- **`supabase/tests/database/rls.test.sql` was extended (not left untouched)** to add `select has_table('public', 'parse_misses', ...)` alongside the other PRD ¬ß7 tables' checks, and `plan(25)` ‚Üí `plan(26)`. The suite's RLS-enabled/policy-count/no-open-policy assertions (checks 1‚Äì6) are already dynamic and would have caught a missing-RLS `parse_misses` automatically with zero edits ‚Äî this addition is belt-and-suspenders schema-shape coverage, matching the brief's explicit ask to "confirm the new table appears in the ... assertion." A dedicated `supabase/tests/database/parse_misses.test.sql` (7 assertions: table exists, both boolean columns default `false`, the per-user dedupe unique index rejects a duplicate `(user_id, raw_sms_hash)` pair with `23505`, and the same positive/negative cross-user RLS pattern as `rls.test.sql`/`new_user.test.sql`) covers what the generic suite can't (column defaults, the dedupe constraint itself, table-specific cross-user isolation).
- **Not executed against the cloud database from this environment** ‚Äî `SUPABASE_DB_URL` isn't set here (same Docker-free/no-DB-credential constraint noted in every prior backend-engineer entry in this file). `npm run test:rls` was confirmed to fail cleanly with its documented "Set SUPABASE_DB_URL..." message rather than a crash. Both new pgTAP files were hand-verified for SQL correctness (constraint names, error codes, FK targets, plan counts matching assertion counts) but need the lead to `supabase db push` the new migration, then run `SUPABASE_DB_URL=‚Ä¶ node supabase/tests/run-pgtap.mjs supabase/tests/database/rls.test.sql` and `... supabase/tests/database/parse_misses.test.sql` against real Postgres before trusting this schema in anger. Exact commands repeated in PROGRESS.md.

## 2026-07-14 ‚Äî Phase 2: Transactions list screen (design-engineer)

**Scope:** `src/routes/{Transactions,Transactions.test}.tsx` (rewritten from the Phase 1 stub); new `src/features/transactions/{transactionGroups,rowSwipe}.ts` (+ tests), `{FilterChip,TransactionRow,RecategorizeSheet}.tsx`; 3 new icons in `src/components/ui/icons.tsx` (`SearchIcon`, `TagIcon`, `TrashIcon`); `src/components/ui/Toast.tsx` extended with an optional inline `action` (+ test). No data-hook files (`queries.ts`/`mutations.ts`/`schemas.ts`/`types.ts`) were modified. Not committed, per brief.

- **`Toast` gained an optional `action: { label, onClick }`.** The brief asked for delete-with-undo; the existing `ToastOptions` shape (`title`/`description`/`variant`/`durationMs`) had no way to attach an action button. Rather than build a bespoke one-off "undo bar" outside the toast system, `ToastItem` now renders an inline coral-600 text button (between the message and the existing dismiss `X`) that runs the action then dismisses ‚Äî the same `stopPropagation` treatment as the dismiss button, so it doesn't double-fire the card's own tap-to-dismiss. Sized like the existing (already-accepted, sub-44px, low-frequency) dismiss control rather than the 44px floor ‚Äî consistent with that existing precedent, not a new deviation. This is a small, generically-useful primitive extension (any future "just did X, tap to undo" moment can reuse it), not a Transactions-specific hack.
- **Undo is a re-add, not a true restore.** The schema has no soft-delete, so "Undo" on a deleted transaction calls `useAddTransaction` with the deleted row's own fields (a fresh id gets assigned server-side) rather than un-deleting the original row. `safeParse`s the reconstructed payload first (mirroring `AddTransactionSheet`'s own defensive pattern) and shows a calm "Couldn't undo that" toast rather than throwing if the schema ever rejects it.
- **Swipe gestures are tested via their pure threshold function, not simulated drag pixels** (`rowSwipe.ts`'s `resolveRowSwipeAction`, unit-tested directly) ‚Äî jsdom has no real pointer/drag geometry, and this exactly mirrors the codebase's own existing precedent (`Sheet.tsx`'s `shouldDismissSheetDrag`, asserted the same way in `Sheet.test.tsx`). The Testing Library coverage for "delete" instead exercises the row's always-present "¬∑¬∑¬∑" actions-menu button, which is simultaneously (a) the actual, required, gesture-free a11y/keyboard/reduced-motion fallback per the brief, and (b) the only realistically automatable path to the same `useDeleteTransaction` call in jsdom. Both paths call the identical `onDelete`/`onRecategorize` handlers, so this isn't testing a different code path than the swipe gesture ‚Äî the gesture only differs in *how* `onDelete`/`onRecategorize` gets invoked (`onDragEnd` vs. an `onClick`), and that dispatch layer is exactly what `rowSwipe.test.ts` covers directly.
- **The swipe-revealed backdrop (a coral-100 "Recategorize" panel behind a left-swipe, an ink-900 "Delete" panel behind a right-swipe... actually left=delete/right=recategorize, see `TransactionRow.tsx`) is decorative only** ‚Äî `aria-hidden`, not focusable or clickable. It exists purely so a swipe *looks* like it's revealing something as the row translates; the swipe only actually commits the action on release, past a threshold (mirroring `Sheet`'s drag-to-dismiss "reveal via elastic drag, decide on release" shape). Making the backdrop itself interactive would create hidden-but-technically-reachable controls sitting fully underneath an opaque foreground at rest ‚Äî a real a11y trap ‚Äî so it deliberately isn't.
- **`FilterChip` (feature-local, not promoted to `components/ui/`) is a minor variant of the existing `CategoryChip`**, differing only in that it doesn't reserve icon space when no icon is passed ‚Äî needed for the text-only "All accounts"/"All categories"/"All" filter chips sitting alongside icon-bearing ones. Kept local to `src/features/transactions/` rather than promoted to a shared primitive since it's a small variant serving one screen today, not a second design-system primitive; if a future screen needs the same "filter row" pattern, promoting it then (once there's a second real caller) is a one-file move.
- **Transactions now mounts its own `TabBar` + `AddTransactionSheet`**, wired to the same global `useUiStore` `activeSheet` Home already uses. Previously only `Home` rendered `TabBar` (every other route was a Phase 1 stub with no nav chrome at all). Since Transactions is now a first-class, fully-functional screen ‚Äî not a stub ‚Äî it needs the thumb-zone Add FAB and tab navigation like any real screen; mounting `AddTransactionSheet` here too (rather than routing "Add" taps back to `/`) is what lets the empty-state's "Add a transaction" button and the TabBar's own Add button actually open the sheet without leaving this screen, per the brief's stated preference ("triggers the add sheet ... if reachable").
- **Not added to `/kitchen-sink`** ‚Äî same reasoning as Home/`AddTransactionSheet` (see the 2026-07-13 entry below): this screen is a live-hook composition needing a real session/data, unlike kitchen-sink's self-contained primitive gallery. Visual QA should hit `/transactions` directly.
- **Account filter matches either `account_id` or `counter_account_id`.** Filtering the list to "M-PESA" should surface a transfer that moved money *out of* M-PESA even though that transfer's own `account_id` might be the Cash side depending on which leg is "primary" ‚Äî matching either side is the more useful interpretation of "show me M-PESA activity."
- **The row's leading glyph for a transfer falls back to `TransactionsIcon`** (the existing generic receipt-shaped glyph, already used in `TabBar`) when the source account hasn't loaded yet, rather than a category glyph (transfers have no category) or a newly-invented "transfer" icon. Once the account resolves, the transfer row shows that account's own icon (`accountIcon`) ‚Äî "money moved out of this account" reads more concretely than a generic symbol.

## 2026-07-14 ‚Äî `handle_new_user` signup trigger (backend-engineer)

**Scope:** `supabase/migrations/20260714090000_create_handle_new_user_trigger.sql`, `supabase/tests/database/new_user.test.sql`. No app code touched; `supabase/seed.sql` left as-is per the task brief.

- **SECURITY DEFINER + `set search_path = ''`, every reference fully schema-qualified.** The trigger fires on `auth.users` insert, before the new user has a JWT/session, so `auth.uid()` is null and the ordinary `user_id = auth.uid()` RLS policies on `profiles`/`accounts`/`categories` would reject every insert. Running as the function owner (the migration role, which owns these tables and ‚Äî per Postgres's default RLS semantics ‚Äî is not subject to their own RLS policies unless `FORCE ROW LEVEL SECURITY` is set, which it isn't here) is what lets the seed succeed at all; this is the standard Supabase idiom for a signup trigger, not a bespoke bypass. Locking `search_path` to empty (rather than leaving it at `public`) closes the classic SECURITY DEFINER hijack vector ‚Äî an attacker who could get a same-named object earlier in a non-empty search_path resolved instead of the intended `public.*` table/type. Every insert hardcodes `user_id = new.id` (the row literally being created in `auth.users`), so this trigger structurally cannot seed data under any user but the one just created.
- **Seed data is a byte-for-byte mirror of `supabase/seed.sql`'s dev-user seed** (3 accounts: M-PESA/mpesa/smartphone, Cash/cash/banknote, Bank/bank/landmark; 20 categories: 5 income + 15 expense, exact name/kind/icon/color/sort_order) ‚Äî verified programmatically (not just by eye) that both files produce identical tuples. `seed.sql` stays local-dev-only (fixed synthetic user, runs only on `db reset`); this trigger is what actually seeds every *real* signup going forward. The two are deliberately duplicated rather than one calling the other, since `seed.sql` seeds a specific fixed user id via a `do $$ ... $$` block and isn't structured as a reusable function ‚Äî a future refactor could extract a shared `seed_default_data(user_id)` helper if this duplication becomes a maintenance burden, but for now the mirror is small and explicit.
- **`on conflict (user_id, name) do nothing` (accounts/categories) / `on conflict (user_id) do nothing` (profiles)**, matching the exact unique constraints already in the accounts/categories/profiles migrations. This makes the trigger resilient to a re-fire (e.g. Supabase Auth retrying an insert) or a profile/accounts row that already exists for some other reason ‚Äî it never errors, never duplicates.
- **`display_name` reads `new.raw_user_meta_data ->> 'name'`, falling through to `null` when absent** ‚Äî per the task's explicit instruction. Email/password signups typically won't populate this; a future Google OAuth signup may populate `name` (or `full_name`, not read here) in `raw_user_meta_data` ‚Äî revisit the exact key once Google sign-in is wired if it turns out to use a different metadata key.
- **pgTAP test (`new_user.test.sql`, plan 5)** inserts a synthetic `auth.users` row inside `begin/rollback` (same proven pattern as `rls.test.sql`'s synthetic users) and asserts: exactly 1 `profiles` row, exactly 3 `accounts` rows, exactly 20 `categories` rows, plus a spot-check each for `M-PESA` and `Food & Groceries`. Self-rolling-back ‚Äî leaves no residue on whatever database (including cloud) it runs against.
- **Not run against cloud from this environment** (Docker-free constraint per prior entries) ‚Äî migration was cross-checked by eyeball plus the programmatic seed-data-equality check above; `npm run check` (179 tests, unaffected ‚Äî no app code touched) confirmed green. The pgTAP execution and `supabase db push` are for the lead to run via `SUPABASE_DB_URL=‚Ä¶ node supabase/tests/run-pgtap.mjs supabase/tests/database/new_user.test.sql`.

## 2026-07-13 ‚Äî Phase 2: manual-entry sheet + Home assembly (design-engineer)

**Scope:** `src/features/transactions/{AddTransactionSheet,iconMaps}.tsx` (+ colocated test), `src/routes/{Home,Home.test}.tsx`, 2 new icons (`CashIcon`, `BankIcon`) appended to `src/components/ui/icons.tsx`. Wired against the already-built data hooks (`useAccounts`/`useCategories`/`useTransactions`/`useAccountBalances`/`useAddTransaction`/`useSafeToSpend`) ‚Äî none of those files were modified. Not committed, per brief.

- **Derived state instead of "reset on open"/"fix up invalid selection" effects.** The first draft used several `useEffect`s (reset form fields when `open` flips true; default the source account once accounts load; fall back off `Transfer` if only one account exists; keep the counter-account distinct from the source) and every one tripped ESLint's `react-hooks/set-state-in-effect` (a real footgun ‚Äî cascading synchronous re-renders ‚Äî not a style nit). Rewritten twice, landing on: (1) the sheet's actual form (`AddTransactionForm`) only mounts while `open` is true (`{open && <AddTransactionForm .../>}`), so every open is a fresh component instance with fresh `useState` initializers ‚Äî no reset effect needed at all; (2) every other "keep in sync" concern (default account, transfer eligibility, category/counter-account validity) is a plain derived value computed at render time from the raw selection + current query data (`accountIdEffective`, `categoryIdEffective`, `counterAccountIdEffective`, `kindEffective`), never written back into state. This is strictly simpler than the effect-based version and has zero risk of the "set state during another component's render" warning that a naive `onChange={parentSetter}` pattern can trip (see next entry).
- **`Keypad`'s `onChange` prop, called from inside its own `setState` updater function, triggers React's "Cannot update a component while rendering a different component" warning when the parent passes its own state setter directly** (e.g. `onChange={setAmountCents}`) ‚Äî this pattern already existed in `KitchenSink`'s `KeypadSection` (`onChange={setCents}`) before this phase, so it isn't new, but it did make the manual-entry sheet's Testing Library tests flaky until the effect-cleanup above landed (once no effect was racing the same render, the warning stopped affecting test outcomes). Flagging here rather than touching `Keypad.tsx` itself, which is Phase 1's signed-off primitive and out of this brief's scope ‚Äî a future pass could move `onChange`'s call to `useEffect`-on-state-change inside `Keypad` itself if the warning needs fully silencing.
- **Category-icon and account-icon mapping (`iconMaps.tsx`) is a small lookup table, not a new prop on `categories`/`accounts`.** `supabase/seed.sql`'s `icon` column holds placeholder strings (`'shopping-basket'`, `'smartphone'`, ...) that don't match the Phase 1 glyph names 1:1 and don't cover every seeded category. `categoryIcon()`/`accountIcon()` map the known strings to the existing Phase 1 glyphs and fall back to `OtherIcon` for anything unmapped (Health, Education, Subscriptions, etc.) ‚Äî consistent with Phase 1 Brief B's own "representative subset, not exhaustive" decision, not a new gap introduced here.
- **Two new icons added (`CashIcon`, `BankIcon`)** so every seeded `account_type` (`mpesa`/`cash`/`bank`/`other`) has a distinct glyph for the account chips. `AirtimeIcon` (already a handset shape) doubles as the M-PESA glyph rather than drawing a new brand-specific icon. `other` falls back to the existing `OtherIcon`.
- **No "last used account" default** ‚Äî the brief allowed "last-used (or first)"; no persistence mechanism for a per-user last-used account exists yet (and CLAUDE.md forbids `localStorage` for anything financial-adjacent), so the sheet defaults the source account to `accounts[0]` ‚Äî the query's own default order (`created_at asc`). A real last-used default is a small follow-up once there's a place to store it (a Zustand UI-store field would be legitimate, since "which account did I use last" is a UI convenience, not financial data, but that's a call for whoever owns that polish pass).
- **A transfer's "To" list simply omits whatever is selected as "From"** rather than allowing the same account in both lists and validating the mismatch after the fact. This makes "counter ‚âÝ source" structurally true from the account chips themselves ‚Äî the zod schema's `counter_account_id !== account_id` check is still the final guard (defense in depth), but the UI can't produce the invalid state in the first place.
- **The Transfer segment disables itself (and silently falls back to Expense) when fewer than two accounts exist**, rather than letting the user pick Transfer and then hitting a dead end. This is an edge case that shouldn't occur past onboarding (three accounts are seeded), but the sheet doesn't assume that always holds.
- **`AddTransactionSheet`/`Home` were NOT added to `/kitchen-sink`.** Kitchen-sink is the primitives gallery ‚Äî every existing entry there is self-contained (no data fetching). Both new pieces are compositions wired to the live Supabase-backed hooks (`useAccounts`, `useCategories`, `useAddTransaction`, `useSafeToSpend`, ...), which need a real auth session to do anything (PROGRESS.md's still-open "Auth gap" ‚Äî there is no sign-in UI yet). Forcing them into kitchen-sink would mean either a permanently-loading gallery cell or bespoke mocking bolted onto a live route, neither of which matches the existing pattern. They're visually verified as the live `/` route instead (with Testing Library covering the mocked-hook states); the lead's 390√ó844 pass should hit `/` directly, not `/kitchen-sink`.
- **Home's "Recent" list resolves a transaction's category name via a `Map` built from `useCategories()`**, joined client-side (`useTransactions` returns bare `category_id`, not a joined name) ‚Äî the same shape the DB/hooks already expose, no new query or view needed.
- **Transfers render as a neutral row** (`tone="default"`, label falls back to `"Transfer"`, no `+`/`-` prefix) and are never included in the income/expense totals anywhere in Home ‚Äî this is inherited directly from `useSafeToSpend`/`calcSafeToSpend`'s own transfer-exclusion, not re-implemented here; the row-level neutrality is purely a display-layer decision so a transfer never *reads* like income or spend even though its `amount_cents` is technically positive like everything else.
- **The safe-to-spend hero's error-retry button calls `queryClient.invalidateQueries()` with no filter** (invalidate everything active), rather than threading through the three underlying query keys (`profile`, `transactions`, `recurringItems`) that `useSafeToSpend` composes internally (which aren't exposed by that hook's public return shape). This is a rare, user-initiated, low-frequency action (a manual "try again" tap), so the blunt approach is an acceptable trade for not reaching into `useSafeToSpend`'s internals from a component that shouldn't know its query-key composition.
- **Test id realism:** `AddTransactionSheet.test.tsx`'s fixture ids had to be corrected from prose-style ids (`'acct-mpesa'`) to RFC-4122-shaped uuids (`'11111111-1111-4111-8111-111111111111'`, version nibble `4`, variant nibble `8`) ‚Äî zod's `.uuid()` validates the actual version/variant bit pattern, not just "8-4-4-4-12 hex" shape, so a naive fake id fails schema validation exactly the way a bug would, which is what first caught the effect-related warnings above (the mutate call was silently short-circuited by `safeParse` failing, surfacing as the wrong toast in the DOM instead of an assertion on the mutate call).

## 2026-07-13 ‚Äî Phase 2: typed data layer, transaction feature folder (feature-engineer)

**Scope:** `src/features/transactions/{types,queryKeys,nairobiDate,queries,mutations,schemas,balanceDelta,useSafeToSpend}.ts` + colocated tests, `src/features/transactions/hooks/useAuthUserId.ts` (+ test), `src/test/supabaseTestHelpers.ts`. No UI/screens (out of scope per brief). Not committed ‚Äî left for the lead to review the money-path seam.

- **`useProfile`/`useSafeToSpend` placement: `src/features/transactions/`, not a `settings/` feature folder.** The brief explicitly said "place it wherever is cleanest... record the placement." `useSafeToSpend` is this folder's primary Phase 2 deliverable and its only consumer of `useProfile` right now; a future settings screen (editing `expected_income_cents`/`cycle_anchor_day`) can import `useProfile` from here rather than the query being duplicated. If a `settings/` feature folder is built later and this placement feels wrong from that side, moving the query is a one-file change (its only import sites will be `useSafeToSpend.ts` and whatever settings screen consumes it).

- **`useUpcomingRecurringBills` filters to `kind: 'expense'` server-side.** The brief's literal instruction was "sum recurring_items whose next_due_date is in [today, periodEnd]" with no kind filter mentioned. `recurring_items.kind` is `income | expense | transfer` (same enum as transactions), so a recurring *income* template (e.g. a modeled recurring salary) or a recurring *transfer* would otherwise inflate `upcomingFixedBillsCents` ‚Äî but `calcSafeToSpend`'s `upcomingFixedBillsCents` param is specifically "fixed bills still due", i.e. money leaving the pool, not money entering it or moving sideways between the user's own accounts. Filtering to `kind='expense'` keeps this consistent with the transfer-exclusion principle CLAUDE.md applies everywhere else. If recurring income templates are added deliberately (e.g. to project future income into safe-to-spend), that would need its own explicit term in `calcSafeToSpend`, not a silent fold into the bills reservation.

- **Missing `profiles` row falls back to `expected_income_cents: 0`, `cycle_anchor_day: 1` rather than blocking `useSafeToSpend` indefinitely.** The schema (per `backend-engineer`'s Phase 0 entry) doesn't guarantee a `profiles` row exists before Home is reachable ‚Äî onboarding is expected to create it (PRD's welcome flow asks for typical income directly), but Phase 2 has no onboarding UI yet. Rather than have `useSafeToSpend` return `data: undefined` forever when a user has no profile (which would render Home's hero permanently blank/loading with no way out), it renders a real, honest number (`KES 0` for the whole period, since declared income is 0) using the same defaults the DB columns themselves default to. This is deliberately visible-and-wrong-until-onboarded rather than invisible-and-stuck ‚Äî matches "never ship a screen without loading/empty/error states" in spirit, since a perpetual loading spinner isn't a real state either.

- **`updateTransactionSchema`'s cross-field invariants (transfer ‚áî `counter_account_id`, transfer ‚áí no `category_id`) only fire when the *same patch* carries the relevant fields together.** A patch is a partial object; you can't validate "transfer requires a counter_account_id" against a patch that only touches `note` without first fetching the row being patched (which the schema layer deliberately doesn't do ‚Äî zod validates shape, not database state). Concretely: changing `kind` to `'transfer'` without also patching `counter_account_id` in the same call is rejected; patching only `counter_account_id` on an existing (already-valid) transfer is allowed through since the schema can't see the untouched `kind`. The DB CHECK constraints (`supabase/migrations/*_create_transactions_table.sql`) remain the final, authoritative guard for any gap this leaves ‚Äî exactly the "defense in depth" the schema/DB pairing is meant to provide, not either one alone.

- **A `crypto.randomUUID()`-based `optimistic-<uuid>` id, not a monotonic counter or `nanoid`,** for the placeholder row an add-mutation inserts into the cache before the server assigns a real id. `crypto.randomUUID` is a Node ‚â•19/all evergreen-browser global ‚Äî no new dependency, no collision risk with real (Postgres `gen_random_uuid()`-style) ids. `isOptimisticId(id)` is exported so a future component can show a pending affordance (e.g. a subtle opacity) on rows not yet confirmed by the server.

- **Balance optimism is delta-based (`balanceDelta.ts`), computed identically for add/update/delete, not "refetch `account_balances` synchronously."** `useUpdateTransaction`/`useDeleteTransaction` reverse the *old* row's delta (found in the cached transaction list, if present) before applying the *new* one ‚Äî this exactly mirrors what the `account_balances` SQL view would compute from the full transaction set, so the optimistic number a user sees mid-flight matches what the server will confirm on `onSettled` invalidation, rather than drifting. `fee_cents` is never touched by this math, matching `backend-engineer`'s existing "fee-modeling decision" entry (a fee is its own separate `expense` row). If the row being updated/deleted isn't in any cached list yet (e.g. a cold cache), the balance patch is skipped rather than guessed ‚Äî the `onSettled` invalidation still corrects it from the server.

- **Test fake for supabase-js's query builder (`src/test/supabaseTestHelpers.ts`) is a `Proxy`-based `chainable()` with zero dependency on `vi`/mocking libraries**, so it's reusable as a plain value even outside `vi.mock` factories. Every chain call is recorded via an `onCall` callback (used to assert e.g. that `insert()` received the session's `user_id`, or that `useTransactions({from,to})` actually applied `.gte`/`.lte` on `occurred_at`). It supports an optional `delayMs` (via `setTimeout`, not a bare microtask) specifically because the DoD's required "optimistic insert, then rollback on error" test needs to inspect the optimistic cache state *before* the fake network call settles ‚Äî a synchronously-resolving fake promise settles faster than `onMutate`'s own `cancelQueries` microtask chain, racing the assertion. `delayMs` makes the fake behave like a real network call (always crosses a macrotask), matching reality rather than being an artifact of the test being "too fast." Each hook test file also has to explicitly flush `useAuthUserId`'s async `getSession()` resolution (`waitForAuthReady()`) before calling `mutate(...)`, or the mutation would run under the `userId: undefined` query-key namespace instead of the authenticated one ‚Äî documented inline in `mutations.test.tsx` since it's an easy trap for whoever adds the next mutation test.

- **`vi.hoisted(() => ({ auth: {...}, from: vi.fn() }))` is duplicated (not shared) across `queries.test.tsx`/`mutations.test.tsx`/`useSafeToSpend.test.tsx`/`useAuthUserId.test.tsx`**, rather than importing one shared hoisted mock object. `vi.hoisted`'s callback runs before other imports are evaluated within that file, so an imported factory function can't reliably be called from inside it without risking a "used before initialization" hazard across files/bundlers ‚Äî the few-line duplication is the safer, standard-idiom trade-off here (Vitest's own docs recommend inlining the hoisted factory).

## 2026-07-13 ‚Äî Phase 2 start: safe-to-spend calculator semantics (lead)

`src/lib/safeToSpend.ts` ‚Äî the pure, exhaustively-tested hero calculator (19 tests; the master prompt reserves the money path for the lead). Two semantic decisions the PRD formula left implicit:

- **`expected_income` = max(declared typical income, income actually received this period).** PRD ¬ß4.5 says "whichever is greater is used, so a bonus raises the number." Implemented as `Math.max` ‚Äî declared income is the floor, real income can only raise the number, never lower it.
- **`fixed_bills_this_period` is passed as bills still DUE later in the period (not yet paid), NOT the full month's bills.** This avoids the classic double-count: a bill already paid is an ordinary `expense` transaction already inside `variable_spend_so_far`; if we also reserved the full month's fixed bills, that paid bill would be subtracted twice. So the caller (the hooks layer, Phase 2) passes only the not-yet-paid remainder as `upcomingFixedBillsCents`. The transactions table has no `recurring_item_id` link in v1, so "which expenses are bill payments" can't be derived from data ‚Äî this split sidesteps that cleanly. Same reasoning for `plannedGoalContributionsCents` (remainder planned, not the whole period).
- **Over-budget is a PERIOD TOTAL, not per-day.** When the pool is negative, `safeToSpendCents` is the negative pool itself (e.g. ‚àí40000), because PRD ¬ß4.5 shows "You're KES X over this **month**" ‚Äî a total. On track (pool ‚â• 0), it's `floor(pool / daysRemaining)` (floored so a fractional cent is never over-promised). `SafeToSpendHero` already keys its calm amber over-state on `safeToSpendCents < 0`.
- **Transfers are excluded structurally, not by discipline** ‚Äî the calc `continue`s on `kind === 'transfer'`, guarded by a test that a huge withdrawal never moves the number. All day/period math is Africa/Nairobi via `@date-fns/tz` `TZDate`, verified with a device-at-22:30-UTC test that correctly reads the next Nairobi day. Non-integer cents throw (no float in the money path).
- **Refunds/reversals are an upstream (parser/data) concern, not the calc's.** The calculator trusts its transaction inputs; a reversed expense must be reconciled away (deleted/offset by `mpesa_ref`) before it reaches here, per the parser's dedupe/reversal handling (Phase 3). The calc's own guarantee is narrower and absolute: transfers never count, and only income/expense within the period up to `now` do.

## 2026-07-13 ‚Äî Phase 0 schema, RLS, derived balances, seed (backend-engineer)

**Scope:** `supabase/migrations/20260713110000`‚Ä¶`20260713111000` (11 files), `supabase/seed.sql`, `supabase/tests/database/rls.test.sql`, `package.json` (`test:db` script).

- **Enums over free-text CHECKs for closed value sets.** `account_type`, `category_kind`, `transaction_kind`, `transaction_source` are Postgres `create type ... as enum`, not `text` + `CHECK (x in (...))`. Rationale: stronger DB-level guarantee (typo in an insert fails at parse time, not silently), self-documenting in `\d table`, and `supabase gen types typescript` turns enums into clean TS union types for free. Trade-off (documented, accepted): adding a new value later requires `ALTER TYPE ... ADD VALUE` in a new migration rather than just relaxing a CHECK ‚Äî acceptable since this value set is genuinely closed (PRD ¬ß7 enumerates it exactly).

- **Fee-modeling decision for `account_balances` (the one the lead flagged to verify).** PRD ¬ß4.2 is explicit: "An M-PESA agent withdrawal is a transfer from M-PESA to Cash; the withdrawal fee is an expense (category: Fees)." That means a withdrawal-with-fee SMS must produce **two transaction rows** at the application/parser layer ‚Äî one `transfer` (M-PESA ‚Üí Cash, full withdrawal amount) and one `expense` (the fee, its own `amount_cents`). Given that, `transactions.fee_cents` (a column PRD ¬ß7 lists on the transaction itself) is **not** authoritative money-movement data ‚Äî it's provenance: what the source SMS/statement literally reported as its fee line, kept for parser debugging, reconciliation display, and the "Fees & Fuliza spotlight" (F10) to cross-check against the actual fee expense row. Consequently `account_balances` **only** ever touches `amount_cents` (via `kind`), and **never subtracts `fee_cents`** ‚Äî doing so would double-count a fee that's already leaving the account through its own expense transaction. This is called out with an inline comment in `20260713111000_create_account_balances_view.sql` and must be respected by the parser (F2) and statement importer (F5): they must emit a separate fee expense row, not rely on `fee_cents` to move money.

- **`account_balances` view uses `security_invoker = true` (PG15+)**, not a `SECURITY DEFINER` function or a materialized view. Rationale: CLAUDE.md requires balances to be *derived, never stored* (a materialized view is a stored/cached balance and would need invalidation logic ‚Äî rejected), and `security_invoker` means the view runs with the querying session's own RLS-restricted privileges rather than the view owner's, so it "just works" under RLS without a bespoke policy mechanism (views can't have their own RLS policies ‚Äî this is the standard idiom for RLS-safe views on Postgres 15+, and the local project's `db.migrations.major_version = 17` supports it).

- **Explicit `grant select, insert, update, delete ... to authenticated` on every table (and `grant select` on the view), in the same migration as the table.** `supabase/config.toml`'s `[api]` section notes the *new* Supabase default does **not** auto-expose newly created `public` schema objects to the Data API roles (`anon`/`authenticated`/`service_role`) without explicit grants ‚Äî this is a change from older Supabase versions. Without these grants, RLS policies would be moot: the querying role wouldn't even have table-level privilege to attempt the row, and API calls would fail with a permissions error unrelated to RLS. RLS policies remain the row-level gate; these grants are the necessary table-level prerequisite.

Rationale: (1) gives `supabase/seed.sql` a stable `ON CONFLICT` target so the seed is idempotent-friendly as the task requires, and (2) prevents a user from silently ending up with two "Cash" accounts or two "Food & Groceries" categories, which would corrupt the mental model the safe-to-spend and insights screens depend on. `categories` uniqueness is on `(user_id, name)` only (not `name, kind`) since the PRD's default category set has no cross-kind name collisions and simpler is safer here.

## 2026-07-14 ‚Äî Phase 3 M-PESA parser core (parser-engineer)

**Scope:** `src/parser/` only (`types.ts`, `money.ts`, `timestamp.ts`, `merchant.ts`, `patterns.json`, `patternTable.ts`, `index.ts`, `__fixtures__/`, and their colocated `*.test.ts`), plus a one-line `tsconfig.app.json` compiler-flag addition (see below). No DB, migrations, components, Edge Functions, or `src/features/` files touched.

- **`resolveJsonModule: true` added to `tsconfig.app.json`.** Not a new dependency ‚Äî a compiler flag needed so `patternTable.ts` can `import` `patterns.json` directly as typed data, which is the whole point of "the pattern table is data" (CLAUDE.md). The alternative (fetch/read the file at runtime) would break the parser's "pure function, no I/O, works offline" requirement and complicate testing for no benefit.

- **`ParsedMpesaMessage.merchant` is a clean DISPLAY string, not the merchant-memory-normalized key.** The contract keeps these separate: `merchant` (trimmed/whitespace-collapsed, casing left as Safaricom sent it, good for the confirmation card UI) vs. `normalizeMerchant(merchant)` (uppercase, trailing phone/code stripped, the actual `merchant_rules` lookup key). Baking normalization into the parsed output would lose display quality for no gain, since the feature-hook layer already has to call `normalizeMerchant` itself before touching `merchant_rules` (that DB read/write is explicitly out of this agent's scope).

- **Withdrawal/deposit/Fuliza/M-Shwari-KCB are all modeled as `kind: 'transfer'`, never `expense`/`income`**, per CLAUDE.md's money rule 3 and the task's explicit "Fuliza drawdown is not income" constraint. `feeCents` rides alongside every family (0 when the SMS mentions none) as a SEPARATE amount the integrator turns into its own "Fees & Fuliza charges" expense row ‚Äî the same generic fee-splitting rule serves withdrawal fees, PayBill/Buy-Goods/Send transaction costs, and the Fuliza access fee uniformly, rather than one-off withdrawal-specific logic.

- **Fuliza and M-Shwari/KCB M-PESA transfers bucket under `counterAccountHint: 'bank'`.** v1's account model (PRD ¬ß4.1) only has M-PESA/Cash/Bank ‚Äî there's no dedicated "loan"/"savings-product" account type ‚Äî so both a Fuliza overdraft facility and an M-Shwari/KCB savings sweep are modeled as the closest fit, "bank". Flagging for whoever owns the account model: if v2 ever adds a dedicated Fuliza/M-Shwari account type, `counterAccountHint` gains a 4th enum value and `patterns.json`'s `fuliza_drawdown`/`fuliza_repayment`/`mshwari_kcb_transfer` entries get a one-line update ‚Äî the rest of the contract is unaffected.

- **Reversal is the one `kind: 'transfer'` family exempted from requiring `counterAccountHint`/`transferDirection`.** A reversal has no real second account of the user's ‚Äî it's a correction, not a movement between two of their accounts ‚Äî so `kind: 'transfer'` there is purely a safety marker meaning "never book this as plain income/expense." The real signal is `reversalOfRef` (the ORIGINAL transaction's `mpesa_ref`), which the integrator uses to look up and negate that transaction. This relaxation is enforced (and tested) in `parsedMpesaMessageSchema`'s `superRefine` and in `patternTable.test.ts`. Full row-splitting rule for every family is documented as a big doc comment at the top of `types.ts` ‚Äî read it before wiring the integration layer.

- **Withdrawal/deposit fee rows need a DISTINCT `mpesa_ref` from the transfer row**, since `transactions.mpesa_ref` is unique per user (PRD ¬ß7) and one SMS parse can produce two rows. Recommended convention (documented in `types.ts`, not enforced in code since row-splitting itself is the integrator's job): `mpesaRef` for the transfer, `` `${mpesaRef}-FEE` `` for the fee expense. This keeps re-parsing the same SMS a true no-op for both rows.

- **Pochi la Biashara vs. plain "sent to person" is a documented, real-world ambiguity ‚Äî not fully solved.** On the PAYING customer's own confirmation SMS, a Pochi la Biashara payment and a normal person-to-person send are usually textually identical (Safaricom doesn't consistently brand Pochi payments differently on the sender side). The `pochi_la_biashara` pattern here matches on a `"- POCHI LA BIASHARA"` marker in the recipient name, which is a plausible-but-not-guaranteed real-world shape; when that marker is absent, a Pochi payment silently and safely falls through to `sent_to_person` (still correctly an expense, just uncategorized/mis-familied) rather than being mis-parsed as something worse. Flagged in the final report as a fallback-log candidate: real Pochi la Biashara message samples should be collected to confirm/replace this heuristic.

- **Corpus scoring counts `rawText` out of the 15 scored fields deliberately** (`corpus.test.ts`): it's an echo of the input the extractor already had, not something extracted, and including it would inflate the accuracy percentage without testing anything. `status` (matched vs. unmatched) is scored as its own field on top of the 15, so a matched-but-wrong-family fixture doesn't silently look identical to a correctly-unmatched one in the aggregate score.

- **Money/timestamp/merchant are three separate single-purpose modules** (`money.ts`, `timestamp.ts`, `merchant.ts`) rather than one grab-bag `utils.ts`, matching the task brief's explicit module boundaries and CLAUDE.md's "pure logic in `src/lib/`"-style discipline applied within `src/parser/` ‚Äî each is independently unit-tested with its own edge-case suite (money: decimal-padding/thousands-separators/prefix-casing/never-parseFloat; timestamp: AM/PM boundary, leap years, invalid-calendar-date rejection, day-boundary UTC conversion; merchant: whitespace/case/trailing-code normalization).

- **`TZDate#toISOString()` renders with the zone's own offset (e.g. `"...+03:00"`), not `"Z"`.** `parseMpesaTimestamp` goes through a plain `new Date(zoned.getTime()).toISOString()` at the very end so the contract's promised "correct ISO instant" is always the standard UTC `"...Z"` form the rest of the app (and `Date.parse`) expects, while all the wall-clock arithmetic that produces `zoned` still happens correctly in Africa/Nairobi via `TZDate`.

Switched to a **GET** target (`action: '/'`, `params: {title,text,url}`): the browser puts the shared SMS straight into `/?text=‚Ä¶`, Home reads it (`extractSharedText`), opens the paste-parse sheet, and strips the params in the close handler. Both approaches end with the text in a GET URL anyway (the SPA must read it somehow), so GET is strictly simpler with the same privacy profile. Home derives the shared text (no `setState`-in-effect) and clears the URL params on close.
